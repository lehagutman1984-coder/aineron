# Сравнительный анализ: KopirkaCMS v4.3 против Aineron Studio
## Стратегическое решение по технологии клонирования сайтов

**Документ для принятия решения go/no-go.**
Автор: архитектура. Аудитория: CTO, продуктовая и юридическая команды.
Связанные документы: `ANALYSIS.md` (полный разбор KopirkaCMS), `ROADMAP.md`.

## 1. Введение — в чём фундаментальная разница

KopirkaCMS и Aineron Studio решают внешне похожую задачу («сделай мне сайт по чужому URL»), но делают это в **двух принципиально несовместимых парадигмах**. Понимание этой разницы — ключ ко всему решению, поэтому начнём именно с неё, не сглаживая.

### KopirkaCMS = СТАТИЧЕСКАЯ КОПИЯ (byte-for-byte capture)

KopirkaCMS — это «фотоаппарат». Он скачивает оригинал целиком и сохраняет его как есть:

- Скачивает **все** ассеты: HTML, CSS, JS, изображения, шрифты, видео — параллельно через `curl_multi_init()` батчами по 20 потоков.
- Переписывает все URL на относительные локальные пути (`replaceMap`, отсортированный по убыванию длины строки во избежание частичных совпадений).
- «Уникализирует» контент — регулярными выражениями удаляет следы CMS, аналитики, og-теги, source maps, переименовывает классы (Python-слой), чтобы уйти от детекторов плагиата.
- Отдаёт **точную визуальную реплику** оригинала. Все файлы сохранены, вёрстка идентична пиксель в пиксель.
- Технологически: PHP 7.4 в процедурном стиле, MySQL, Apache, без ООП и MVC.

**Результат: визуально неотличимая копия целевого сайта. Но это мёртвый слепок — без реальной серверной логики, без работающего бэкенда.**

### Aineron Studio = AI-РЕГЕНЕРАЦИЯ (functional re-creation)

Aineron Studio — это «художник по описанию». Он смотрит на оригинал, понимает *что это за продукт*, и пишет **новый код с нуля**:

- Конвейер `Architect → Coder → Guardian` генерирует свежий React/Next.js/Vite-код.
- На выходе — **новое функциональное приложение**, которое приближает оригинал по смыслу, но **не является попиксельной копией**.
- Вся визуальная точность (точный CSS, изображения, раскладка) **теряется на этапе обработки краула**.

Важная фактическая поправка (и она усиливает наш план, поэтому фиксируем её точно). Распространённое утверждение «вся визуальная точность отбрасывается на этапе краула» — **не совсем верно**. Сам краулер уже сейчас извлекает HTML и ссылки на CSS:

```python
# src/studio/crawler.py — crawl()
return {
    'html': r.text,                       # ← полный HTML здесь есть
    'title': (soup.title.string if soup.title else ''),
    'text': soup.get_text(separator='\n', strip=True)[:20000],
    'css_links': [link.get('href') for link in soup.find_all('link', rel='stylesheet')],
}
```

**Отбрасывание происходит на один слой позже** — в Celery-задаче, которая сохраняет результат краула в проект. Именно здесь из богатого payload остаётся только заголовок и обрезанный текст:

```python
# src/studio/tasks.py — crawl_and_analyze() / crawl_spa_task(), строки ~1045-1048
project.interview_data['crawled'] = {
    'title': data['title'],
    'text': data['text'][:8000],          # ← HTML, css_links, вёрстка выброшены здесь
}
```

Далее `ArchitectAgent._build_context()` забирает из сохранённого ещё меньше — `text[:3000]`:

```python
# src/studio/agents/architect.py — _build_context()
crawled = (self.project.interview_data or {}).get('crawled', {})
if crawled:
    parts.append(
        f"\nCrawled site:\nTitle: {crawled.get('title', '')}\n"
        f"{crawled.get('text', '')[:3000]}"
    )
```

Даже `ScreenshotAgent` (vision-режим) возвращает лишь **текстовое описание** макета, а не пиксельные данные.

Этот нюанс — не педантизм. Он делает точку внедрения нового режима **конкретной и реалистичной**: мы перехватываем уже скачанный HTML-payload, который сейчас просто выбрасывается, а не пристраиваем отдельную систему загрузки. Это сильнейший аргумент в пользу выбранного варианта (см. §6).

### Главный вывод введения

Эти два инструмента **не находятся на одной оси сравнения**:

- KopirkaCMS выигрывает в **попиксельной точности статического захвата**.
- Aineron Studio выигрывает в **функциональной регенерации** (формы, логика, роутинг, API, поддерживаемый код).

Поэтому вопрос «кто лучше» поставлен некорректно. Правильный вопрос: **стоит ли добавить Studio способность, которой у неё сейчас нет** — точный статический захват. Это **аддитивное** решение, а не замена.

## 2. Что делает KopirkaCMS лучше (с конкретными алгоритмами)

Ниже — реально ценные инженерные решения KopirkaCMS, которые у Studio отсутствуют. Это и есть кандидаты на портирование.

### 2.1. Полный конвейер скачивания ассетов

Главная сила. KopirkaCMS захватывает оригинал целиком:

- **Параллельная загрузка** через `downloadFilesParallel($jobs, $base, 20, $statusFile)` — `curl_multi` батчами по 20 потоков. 100+ ассетов скачиваются за секунды вместо минут.
- **Рекурсивная локализация CSS** — `localizeCssAssets()` заходит внутрь CSS-файлов, скачивает всё из `url(...)` и обрабатывает цепочки `@import` рекурсивно. Это критично: половина шрифтов и фоновых картинок прячется именно внутри CSS.
- **Извлечение из DOM + 13 lazy-load атрибутов**: `data-src`, `data-original`, `data-lazy`, `data-bg` и др., плюс `srcset`, `<picture>`, `<video poster>`, фоновые изображения из inline-стилей.
- **Универсальный regex-fallback** по сырому HTML — ловит URL ассетов в `__NEXT_DATA__` и подобных JSON-блобах по расширению файла, без разбора JSON.

У Studio этого **нет вообще** — ни одного скачанного байта ассетов.

### 2.2. Карта переписывания путей (replaceMap)

```php
uksort($replaceMap, fn($a,$b) => strlen($b) - strlen($a)); // длинные строки первыми
foreach ($replaceMap as $orig => $local) {
    $rewrittenHTML = str_replace($orig, $local, $rewrittenHTML);
}
```

Сортировка по убыванию длины — простой, но важный трюк: предотвращает частичную замену, когда короткий URL является префиксом длинного. Готовый алгоритм для портирования.

### 2.3. Многоуровневый fallback краула (5 уровней)

KopirkaCMS умеет адаптироваться под сложность сайта:

1. cURL (Chrome UA) — статика, SSR.
2. `looksEmptySpa()` — эвристика авто-детекта пустого SPA-каркаса.
3. Scrape.do (платный headless-рендер как сервис) — для CSR-SPA.
4. Python/requests — альтернативный движок.
5. Selenium ChromeDriver — для сайтов за Cloudflare/DataDome.

У Studio fallback **одноуровневый**: `requests` → Playwright (по условию «текст < 200 символов»).

Эвристика `looksEmptySpa()` особенно ценна — детектит SPA по нескольким признакам (размер HTML < 4 КБ; мало текста + нет картинок; много `<script>` при коротком тексте) без внешних сервисов.

### 2.4. Standalone ZIP-экспорт

Уникальная фича: экспорт клона в самодостаточный ZIP с мини-админкой, **работающей без MySQL** (всё через `data.json` / `submissions.json`). Клон деплоится на любой PHP-хостинг одним кликом. Сам алгоритм упаковки (HTML + css/js/img/fonts + служебные файлы) — ценный паттерн.

### 2.5. Спорные/«серые» фичи (НЕ портируем — см. §5)

Для полноты картины перечислим, но сразу маркируем как **исключаемые по юридическим причинам**:

- **Перехват форм** (`tilda-form-interceptor.js` — XHR monkey-patch; `bind_form.php` — подмена action).
- **Смарт-редиректы** по device/OS/country (таблица `redirects`).
- **Уникализация для обхода детекторов плагиата** (цель — уйти от обнаружения копирования).

### 2.6. Сводка сильных сторон (с оценками из ANALYSIS.md)

| Возможность | Реализация | Оценка |
|-------------|-----------|--------|
| Клонирование статических сайтов | cURL + DOMDocument | 5/5 |
| Параллельная загрузка ассетов | curl_multi, 20 потоков | 5/5 |
| Рекурсивный CSS-парсер | `localizeCssAssets()` | 5/5 |
| Lazy-load (13 атрибутов) | data-src и др. | 5/5 |
| Авто-детект SPA | `looksEmptySpa()` | 4/5 |
| Universal regex-fallback ассетов | по расширению | 4/5 |
| Standalone ZIP-экспорт | `export_site.php` | 5/5 |

## 3. Что делает Aineron Studio лучше

### 3.1. Генерирует ФУНКЦИОНАЛЬНЫЕ приложения, а не статику

Это категориальное преимущество. Studio на выходе даёт работающее приложение с формами, логикой, роутингом, API-вызовами, моками данных — а не мёртвый HTML-слепок. KopirkaCMS не способен клонировать SPA с реальным бэкендом; он копирует только то, что видит браузер.

### 3.2. Современный, поддерживаемый стек

Вывод — React / Next.js / Vue / Vite / Telegram Mini App (`STACK_CHOICES` в `models.py`). Это деплоится на Vercel/Timeweb/Selectel, читается людьми и развивается дальше. Клон KopirkaCMS — это чужой минифицированный bundle, который невозможно осмысленно поддерживать.

### 3.3. Полноценный AI-конвейер с код-ревью

`Architect → Coder → Guardian` с итеративной проверкой кода, `StudioPipelineState` (контроль зацикливания, повторов ошибок, autofix), бюджет в «звёздах». Это инженерный продукт, а не скрипт.

### 3.4. Гибкие режимы входа

`ENTRY_CHOICES = [('description', 'С нуля'), ('clone_url', 'Клон по URL')]` — генерация с нуля по описанию **и** клонирование-вдохновение по URL.

### 3.5. Screenshot-to-code

`ScreenshotAgent` с vision-моделью описывает макет по скриншоту для последующей генерации кода.

### 3.6. Асинхронность и стриминг прогресса

Celery-обработка, SSE-стриминг прогресса на фронтенд (`publish_event`), выделенные очереди (отдельная `PLAYWRIGHT_QUEUE` под prefork-воркер, остальное — на gevent).

### 3.7. Инкрементальная генерация в стиле git-патчей

EditBlock / `file_delta` — патч-коммиты для файлов любого размера (см. MEMORY: Phase 8).

### 3.8. Слабые стороны Studio (честно)

- Не воспроизводит визуальную точность — нет ни одного скачанного ассета.
- Краул сохраняет только текст, не структуру/раскладку (см. §1).
- Конвейер загрузки ассетов отсутствует как класс.
- Fallback краула одноуровневый (нет Scrape.do, нет Selenium).
- Сгенерированный код приближает *концепцию* сайта, а не его *дизайн*.

## 5. Юридические и этические риски (ОБЯЗАТЕЛЬНЫЙ РАЗДЕЛ)

Этот раздел — не формальность. Часть функций KopirkaCMS относится к grey-/black-hat и создала бы **серьёзный юридический и репутационный риск** для легитимного SaaS уровня aineron.ru. Эти же три функции мы **исключаем из объёма работ** в §7 — раздел и рекомендация образуют единый список.

### 5.1. Перехват форм = присвоение чужих лидов

`tilda-form-interceptor.js` через monkey-patch `XMLHttpRequest.prototype.send` перехватывает данные форм на клонированном сайте и параллельно отправляет их на сторонний сервер. Оригинальная отправка не блокируется — работают оба канала.

**Суть:** захват лидов, которые по праву принадлежат владельцу оригинального сайта. В большинстве юрисдикций это с высокой вероятностью **незаконно** — несанкционированный доступ к данным / неправомерный сбор персональных данных. Для РФ это прямое нарушение 152-ФЗ (обработка ПДн без законного основания и согласия субъекта на стороне, не являющейся оператором).

> **Исключается из scope `clone_exact`** именно по этой причине.

### 5.2. Смарт-редиректы = перехват чужого трафика

Таблица `redirects` позволяет после клонирования перенаправлять посетителей (по device/OS/country) на сторонний оффер. Это **увод трафика**, принадлежащего оригиналу, и может квалифицироваться как недобросовестная конкуренция и введение пользователей в заблуждение.

> **Исключается из scope `clone_exact`.**

### 5.3. Уникализация для обхода детекторов плагиата = умысел на уклонение

`uniquifyHTML/CSS/JS`, переименование классов, удаление маркеров источника, перекодирование изображений ради смены хеша — всё это имеет **явно декларированную цель**: пройти детекторы плагиата и скрыть факт копирования. Само наличие такой цели — отягчающий фактор в спорах об авторском праве: это документированный умысел на уклонение от обнаружения нарушения.

> **Исключается из scope `clone_exact`.**

### 5.4. Общая позиция

- Статический захват сам по себе (скачать публично доступный HTML/CSS) — инструмент двойного назначения и допустим как технология для **легальных сценариев**: перенос *собственного* сайта, архивация по разрешению, прототип для клиента по его материалам.
- Поэтому `clone_exact` должен быть оформлен как инструмент **захвата с согласия/правом** (по аналогии с тем, как Studio уже использует `is_safe_url` как технический guard, нужен и **юридический guard**: дисклеймер об ответственности, чек-бокс подтверждения прав, отказ обслуживать явные нарушения).
- Три функции из §5.1–5.3 не переносятся **никогда**, даже как опция.

## 6. Стратегическая рекомендация

Рассмотрим все пять вариантов и вынесем вердикт по каждому.

### Разбор вариантов

**A) Заменить клон Studio подходом KopirkaCMS — ОТКЛОНЕНО.**
Это уничтожило бы главное преимущество Studio (функциональная регенерация) ради одной оси (попиксельность). Замена там, где нужно дополнение. Регресс продукта.

**B) Встроить PHP-код KopirkaCMS в Django (запуск PHP как subprocess) — ОТКЛОНЕНО.**
Запуск PHP 7.4 (архитектура 4/10, безопасность 3/10, `exec()`, отключённый SSL, хардкод-ключи) как подпроцесса внутри Django — это операционная и security-деградация. Два рантайма, чужой небезопасный код в продакшене, зависимость от сервера лицензий `cmskopirka.pro`. PHP остаётся **только референсной реализацией, в продакшен не попадает.**

**D) Держать оба полностью раздельно — ОТКЛОНЕНО.**
KopirkaCMS как отдельный продукт не приносит синергии, тащит юридический хвост и дублирует инфраструктуру. Пользователь aineron.ru не получает единого опыта.

**C) Портировать ценные алгоритмы в Python/Celery как новый `entry_mode='clone_exact'` — ВЫБРАНО (основа).**

**E) Гибрид: статический захват + передача результата в Studio для AI-доработки — ВЫБРАНО (фаза 2).**

### Вердикт

**Рекомендуется вариант C как конкретный «хребет» решения, с вариантом E как естественной второй фазой.**

Конкретно:

1. **Фаза 1 (вариант C).** Портировать **конвейер ассетов** (параллельная загрузка, рекурсивная локализация CSS, переписывание путей, ZIP-экспорт) в `crawler.py` и новую Celery-задачу как режим `entry_mode='clone_exact'` рядом с существующим `clone_url`. Результат — точная статическая копия, разворачиваемая через статический сервер.

2. **Фаза 2 (вариант E).** Скормить захваченную структуру и ассеты существующему конвейеру `Architect → Coder`, чтобы AI **улучшал** точную копию (переписывал её в поддерживаемый React/Next.js, добавлял функциональность). Так мы соединяем попиксельную точность KopirkaCMS с функциональной регенерацией Studio.

3. **Scope-out (из §5).** Функции перехвата форм, смарт-редиректов и уникализации-для-обхода-детекторов **не переносятся** — по юридическим причинам.

4. **PHP = референс.** Исходники KopirkaCMS используются как справочная реализация алгоритмов, в деплой не попадают.

### Почему C, а не A

Потому что преимущество — **аддитивное**. Studio получает способность, которой у неё нет (точный статический захват), не теряя той, что есть (функциональная регенерация). Точка внедрения уже существует: HTML сейчас скачивается в `crawl()` и выбрасывается в `crawl_and_analyze()` (§1). Мы перехватываем уже имеющийся payload — это минимальный, точечный, обратимый разрез по системе.

### 7.2. Изменения в модели данных

**Новый `entry_mode`** в `src/studio/models.py`:

```python
ENTRY_CHOICES = [
    ('description', 'С нуля'),
    ('clone_url', 'Клон по URL'),       # AI-вдохновение (как есть)
    ('clone_exact', 'Точная копия'),    # НОВОЕ: статический захват
]
```

**Хранение бинарных ассетов — важное архитектурное решение.** Текущая модель `StudioFile.content = TextField` **не подходит** для изображений и шрифтов (бинарные данные не помещаются в text-поле). Нужно одно из:

- новая модель `StudioAsset(project FK, path, file=FileField(...), content_type, size)` поверх Django storage; **или**
- хранение ассетов в файловой системе/объектном хранилище per-project (директория вида `studio/clones/<project_id>/`), а в БД — только манифест путей.

**Рекомендация:** `FileField`/Django storage (S3-совместимое в проде). `StudioFile` остаётся только для текстовых артефактов (HTML/CSS/JS), бинарь — в новой сущности. Это решение нужно зафиксировать явно перед началом работ.

### 7.3. Изменения в `crawler.py`

Добавить функцию полного захвата (не ломая существующие `crawl` / `crawl_spa`):

```python
def crawl_exact(url: str, *, render_js: bool = False) -> dict:
    """
    Полный статический захват: HTML + все ассеты + переписанные пути.
    Возвращает: {html, assets: [{path, bytes/url, content_type}], rewrite_map}.
    """
    if not is_safe_url(url):
        raise ValueError('Небезопасный URL (SSRF)')
    # 1. fetch HTML (requests или Playwright при render_js)
    # 2. DOM + regex-fallback → список URL ассетов (+ 13 lazy-атрибутов)
    # 3. download_assets_parallel(urls)  ← порт downloadFilesParallel
    # 4. localize_css_assets(css)        ← порт localizeCssAssets (рекурсивно)
    # 5. build_rewrite_map(...)          ← порт replaceMap (сорт по длине desc)
    # 6. вернуть структуру
```

**SSRF — расширенная поверхность атаки.** Сейчас `is_safe_url` вызывается один раз на входной URL. В `clone_exact` мы скачиваем **десятки под-ресурсов**, каждый из которых — отдельный вектор SSRF. `is_safe_url` обязан вызываться **на каждый** URL ассета перед загрузкой, а не только на entry-URL.

**Параллелизм:** `curl_multi` (20 потоков) → `ThreadPoolExecutor(max_workers=20)` либо `asyncio + aiohttp`. Ограничить число воркеров и общий таймаут.

### 7.4. Новые Celery-задачи и выбор очереди

В `src/studio/tasks.py`:

```python
@shared_task(bind=True, max_retries=2, queue=QUEUE)   # gevent-очередь: I/O-bound
def clone_exact_task(self, project_id):
    """Точный захват: качаем HTML+ассеты, переписываем пути, сохраняем как StudioFile/StudioAsset."""
    ...

@shared_task(bind=True, queue=QUEUE)
def export_clone_zip(self, project_id):
    """Упаковать захваченный клон в самодостаточный ZIP (порт export_site.php, без мини-админки/MySQL)."""
    ...
```

**Маппинг очередей** (важно для стабильности):

- Скачивание ассетов — **I/O-bound**, идеально ложится на **gevent-воркер** (основная `QUEUE`).
- JS-рендер (если нужен `render_js`) — **только prefork** `PLAYWRIGHT_QUEUE`, как и существующий `crawl_spa_task` (Playwright нельзя запускать в gevent).

То есть `clone_exact_task` для статики идёт на gevent-очередь; ветка с рендерингом делегирует в `PLAYWRIGHT_QUEUE` ровно по тому же паттерну, что уже есть в `crawl_and_analyze → crawl_spa_task`.

### 7.5. Точка внедрения (минимальный разрез)

Сейчас результат краула выбрасывается здесь:

```python
# tasks.py — crawl_and_analyze(), ~1045
project.interview_data['crawled'] = {'title': data['title'], 'text': data['text'][:8000]}
```

Для `entry_mode='clone_exact'` маршрутизируем в новый путь **до** этого отбрасывания: вместо сжатия в текст — сохраняем полный захват через `clone_exact_task`. Существующий путь `clone_url` остаётся нетронутым.

### 7.6. Превью и отдача результата (отличие от текущего флоу)

Это ключевое расхождение с существующим конвейером. Сейчас Studio поднимает dev-сервер в sandbox-контейнере (`sandbox_container_id`, `preview_port=3000`) — это нужно для Vite/Next dev-режима AI-генерируемого приложения.

**Статический клон — это просто HTML/CSS/JS.** Его НЕ нужно (и нельзя) гонять через Vite/Next dev-сервер. Для `clone_exact` нужен **статический файловый сервер** (отдача из storage/директории клона), а не dev-sandbox. Это сознательное расхождение с текущим `preview_port`-флоу — фиксируем как отдельную задачу инфраструктуры.

### 7.7. Фаза 2 (вариант E) — мост в AI-конвейер

После точного захвата дать пользователю выбор: «оставить как точную копию» **или** «улучшить через AI». Во втором случае захваченные HTML/CSS и манифест ассетов передаются в `ArchitectAgent` как богатый контекст (уже не `text[:3000]`, а реальная структура), и далее `Coder → Guardian` переписывают копию в поддерживаемый React/Next.js. Так попиксельная точность становится **отправной точкой** для функциональной регенерации.

### 7.8. Юридические guard'ы (из §5.4)

- Чек-бокс подтверждения прав на захват перед запуском `clone_exact`.
- Дисклеймер об ответственности пользователя.
- Никакого перехвата форм, редиректов и anti-plagiarism-уникализации — ни в каком виде.
- `is_safe_url` на каждый ассет.

### 7.9. Объём и порядок работ

1. Модель: `entry_mode='clone_exact'` + сущность хранения бинарных ассетов (миграция).
2. `crawler.py::crawl_exact` + порт `download_assets_parallel`, `localize_css_assets`, `build_rewrite_map`, `looks_empty_spa`.
3. `tasks.py::clone_exact_task` (маршрутизация очередей gevent/Playwright) + точка внедрения.
4. Статический превью-сервер для клонов.
5. `export_clone_zip` (порт ZIP-экспорта, без мини-админки/MySQL).
6. Фронтенд: режим `clone_exact`, юр. чек-бокс, кнопка «улучшить через AI» (мост в фазу E).
7. Тесты: SSRF на ассетах, переписывание путей, рекурсия CSS.
