# ROADMAP aineron.ru — План развития до топ-3 AI-сервисов России

## Vision

Aineron.ru становится двуглавой платформой: **лучший в России веб-чат для конечных пользователей** (B2C) И **OpenAI-совместимый API для разработчиков** (B2B/dev). Конкурент routerapi.ru силён только во втором, у него нет веб-чата, видео, триала и каталога. Мы забираем его рынок разработчиков, не теряя своего преимущества в массовом сегменте.

Стратегический фундамент: единый DRF-слой `/api/v1/`, который одновременно обслуживает (1) новый Next.js фронтенд, (2) внешних разработчиков и (3) IDE-интеграции. Один контракт — три потребителя. Это означает, что качественно построенная ФАЗА 1 (API) автоматически де-рискует ФАЗЫ 2–3 (миграция на Next.js): фронтенд получает готовый, документированный, версионированный backend вместо россыпи ad-hoc AJAX-вьюх.

**Цель по срокам:** 6 фаз, ~6–9 месяцев. Принцип приоритизации — от быстрых побед (ФАЗА 0, дни) к долгосрочным преимуществам (ФАЗА 5–6, кварталы).

## Стратегия (как обгоняем)

1. **Не копировать — поглощать.** Реализуем полный OpenAI/Anthropic-совместимый API (паритет с конкурентом), но поверх него оставляем то, чего у конкурента нет: чат, видео, триал, каталог. Разработчик, который пришёл за API, видит ещё и playground, и каталог — конверсия выше.
2. **Единый API-слой как мультипликатор.** `/api/v1/` строится один раз и обслуживает Next.js, внешних devs и IDE. Это снижает совокупную стоимость ФАЗ 2–4.
3. **Гибридный биллинг.** Web-чат остаётся на модели «звёзды за сообщение» (привычно массовому пользователю). Dev-API вводит **токенное метрирование** с конвертацией токенов в звёзды/рубли — это даёт паритет с pay-as-you-go конкурента, не ломая B2C-экономику. Детали — в разделе «Архитектура API-слоя».
4. **Русскоязычное SEO-доминирование.** Блог конкурента на английском — это его слабость на рынке РФ. Мы строим 50+ технических статей на русском + структурированные данные (JSON-LD) на каждой странице модели. Цель — органика по запросам «GPT-5 API без VPN», «Claude API Россия» и т.д.
5. **Профессиональный, строгий UI.** Никаких эмодзи нигде в интерфейсе. Только векторные иконки (Lucide React). Это позиционирует нас как serious tool для разработчиков, а не как «игрушку».

## ФАЗА 1: API для разработчиков (DRF-слой)

**Цель:** Построить версионированный `/api/v1/` на Django REST Framework с OpenAI-совместимыми эндпоинтами, API-ключами и токенным биллингом. Это и есть будущий backend для Next.js, и продукт для внешних разработчиков. Достигаем паритета с routerapi.ru по dev-функционалу.

**Метрики успеха фазы:**
- `POST /api/v1/chat/completions` проходит тест официальным `openai` Python SDK (только смена `base_url` и `api_key`).
- Стриминг (SSE) работает с `stream=true`.
- Каждый вызов корректно списывает звёзды по токенному метру; при ошибке апстрима — возврат.
- Rate-limiting и аутентификация по `Authorization: Bearer ak_...` работают.
- OpenAPI-схема генерируется автоматически и валидна.

1. **НИКАКИХ ЭМОДЗИ нигде в интерфейсе** — ни в JSX/TSX, ни в строках, ни в данных, ни в email-шаблонах, ни в Django-шаблонах. Это требование закреплено линтером (`frontend/.eslintrc.json`, ФАЗА 2) и CI-скриптом для Python/шаблонов (`scripts/check_no_emoji.py`, ФАЗА 0). Существующие эмодзи (`⭐` в `aitext/models.py:37`, эмодзи в `users/models.py`, `index.html`, `referral.html`, email-шаблонах, `constructor.js`, `profile.js`, `stars_modal.js`) удаляются в ФАЗЕ 0 — это не гипотетическая, а подтверждённая грепом задача.
2. **Только профессиональные векторные иконки.** Единственная разрешённая библиотека — **Lucide React**. Все иконки идут через обёртку `<Icon name=... size=.../>`. Heroicons/Phosphor допустимы как запасной вариант, но в одном проекте — одна библиотека (Lucide). Никаких растровых иконок и эмодзи-заменителей.
3. **Звёзды (валюта) — иконкой, не эмодзи.** Внутренняя валюта отображается иконкой `Star` из Lucide + число, никогда символом `⭐`.

### Токены темы (Tailwind v4, CSS-переменные)
- **Палитра:** нейтральная база (оттенки серого/zinc), один акцентный цвет (например, indigo/blue) для действий, семантические цвета (success/warning/danger) приглушённые. Тёмная тема — основная, светлая — опциональная.
- **Типографика:** один гротеск для UI (Inter/Geist), моноширинный (Geist Mono/JetBrains Mono) для кода и API-ключей. Чёткая шкала размеров.
- **Spacing/радиусы:** строгая 4px-сетка, умеренные радиусы (6–8px), без «пузырей».
- **Тени:** минимальные, плоский дизайн; глубина — границами и фоном, не тяжёлыми тенями.

### Структура компонентов
- `frontend/components/ui/` — атомы (Button, Input, Select, Card, Modal, Toast, Badge, Tabs, Tooltip, Spinner, Table, Icon).
- `frontend/components/<domain>/` — доменные блоки (chat, catalog, billing, charts, seo).
- Варианты компонентов через `cva` (class-variance-authority); композиция через Radix UI primitives для доступности (a11y).

### Доступность и качество
- Все интерактивные элементы доступны с клавиатуры, ARIA-атрибуты, контраст WCAG AA.
- Сторибук/визуальные снапшоты для регрессии UI (опционально, ФАЗА 6).

## SEO-стратегия для выхода в топ-3

### Принцип
Главная асимметрия против routerapi.ru — **их блог на английском**. Российская аудитория и поисковики РФ (Яндекс — доминирующий) отдают приоритет русскоязычному релевантному контенту. Мы строим русскоязычную контент-машину + безупречное техническое SEO.

### Три уровня
1. **Техническое SEO (ФАЗА 0 + ФАЗА 5):**
   - JSON-LD на каждом типе страниц: `SoftwareApplication`/`Product` (модели), `Article` (блог), `FAQPage` (FAQ), `BreadcrumbList` (навигация), `Organization` (бренд).
   - Канонические URL, корректная пагинация, чистый sitemap (динамический на Next.js), robots с защитой служебных путей.
   - Core Web Vitals в зелёной зоне (`next/image`, `next/font`, code-splitting, ISR/SSG).
   - SSR/SSG для всего публичного контента — никакого client-only рендера для индексируемых страниц.
2. **Контентное SEO (ФАЗА 5):**
   - Кластер «dev» (паритет с темами конкурента, но по-русски): подключение каждой модели, миграция с OpenAI, примеры кода.
   - Кластер «B2C»: сравнения моделей, гайды по генерации, выбор нейросети.
   - Программные SEO-страницы под длинный хвост по каждой из 38 моделей («{Модель} API без VPN», «{Модель} цена», «{Модель} онлайн»).
   - Внутренняя перелинковка: статьи ↔ каталог ↔ похожие модели.
3. **Семантическое ядро и мониторинг:**
   - Целевые запросы: «GPT-5 API Россия», «Claude API без VPN», «нейросеть для генерации видео онлайн», «ChatGPT без VPN», «OpenAI совместимый API РФ» и длинный хвост.
   - Яндекс.Вебмастер + Google Search Console + Метрика/GA4 для мониторинга позиций и индексации.

### Цель
Топ-3 по 20+ коммерческим/информационным запросам, органический трафик ×3, rich-сниппеты (FAQ-аккордеоны, рейтинги) в выдаче.

## Архитектура API-слоя для разработчиков

### Единый слой `/api/v1/` — три потребителя
DRF-приложение `api` (ФАЗА 1) — **единственный** программный контракт платформы. Его потребляют:
1. **Next.js фронтенд** (ФАЗЫ 2–3) — web-чат, каталог, кабинет.
2. **Внешние разработчики** — через API-ключи `ak_...`.
3. **IDE-интеграции** (Cursor/Cline/Continue) — через OpenAI/Anthropic-совместимость.

Поэтому качественная ФАЗА 1 де-рискует ФАЗЫ 2–4: фронтенд строится не на ad-hoc AJAX, а на версионированном, типизированном (OpenAPI → TS-типы), документированном API. Существующие legacy-вьюхи (`ajax_login`, `ajax_register` и др. в `users/urls_api.py`) поэтапно рефакторятся в этот слой и удаляются (ФАЗА 3).

### Совместимость
- **OpenAI:** `/api/v1/chat/completions` (+ streaming), `/api/v1/models`, `/api/v1/images/generations`, `/api/v1/embeddings`, `/api/v1/audio/*`. Работает сменой `base_url`+`api_key` в официальном `openai` SDK.
- **Anthropic:** `/api/v1/messages`. Работает с Anthropic SDK.
- Маппинг `model` (строка запроса) → `NeuralNetwork.model_name`; проксирование на laozhang.ai через существующий `get_laozhang_client()`.

### Биллинг: мост «звёзды ↔ токены» (ключевое архитектурное решение)
Web-чат и dev-API считают по-разному, но делят **один кошелёк** `CustomUser.pages_count` (звёзды) / `rub_balance` / `Organization.balance_rub`:

- **Web-чат (B2C):** списание «звёзды за сообщение» по `NeuralNetwork.cost_per_message` — без изменений, привычно массовому пользователю.
- **Dev-API:** **токенное метрирование**. Новое поле `NeuralNetwork.stars_per_1k_tokens` (default выводится из `cost_per_message`). После каждого вызова сервис `charge_for_tokens()` берёт `usage.total_tokens` из ответа апстрима, конвертирует в звёзды и списывает; пишет запись `TokenUsage` (источник аналитики и закрывающих документов). Это даёт **паритет с pay-as-you-go конкурента**, не ломая B2C-экономику звёзд.
- **Возврат при ошибке:** при сбое апстрима списание не происходит / возвращается (как уже сделано для генерации изображений в текущем коде).
- **B2B:** организационные ключи списывают из `Organization.balance_rub`; расход атрибутируется участнику (`OrganizationMember`) и модели — основа дашборда (ФАЗА 4) и счетов/актов.

### Безопасность и эксплуатация
- Аутентификация: `APIKeyAuthentication` (Bearer `ak_...`, хранится только хеш ключа).
- Лимиты: `ScopedRateThrottle` per-key (RPM/TPM) на Redis.
- Ошибки: OpenAI-формат `{"error":{...}}` с корректными кодами (401/402/429/400) для правильного retry-поведения SDK.
- Документация: `drf-spectacular` → OpenAPI-схема → Swagger UI/Redoc + Postman-коллекция + примеры на 4 языках (curl, Python, Node, PHP).
- Версионирование: `URLPathVersioning` (`/api/v1/`) — пространство для несовместимых изменений без поломки клиентов.

## Технические зависимости и риски

### Зависимости (порядок обязателен)
- ФАЗА 1 (DRF-слой) — **фундамент**. ФАЗЫ 2–4 зависят от неё: Next.js потребляет `/api/v1/`, B2B-биллинг строится на `TokenUsage`/`APIKey`.
- ФАЗА 2 (инфраструктура Next.js) предшествует ФАЗЕ 3 (миграция страниц).
- JSON-LD/SEO из ФАЗЫ 0 должен быть **перенесён**, а не потерян при миграции на Next.js (ФАЗА 3).
- Новые пакеты: `djangorestframework`, `drf-spectacular`, `django-cors-headers` (бэк); `next`, `react-query`, `zustand`, `tailwindcss@4`, `lucide-react`, `openapi-typescript`, `class-variance-authority`, Radix UI (фронт).

### Риски и митигация
- **Регресс SEO при миграции на Next.js.** Митигация: SSR/SSG для всего публичного контента, перенос JSON-LD/мета/canonical 1:1, сравнение позиций до/после, поэтапный strangler-перенос (каталог по одной странице), не удалять Django-шаблон до подтверждения паритета.
- **Расхождение биллинга звёзды↔токены.** Митигация: единый кошелёк, журнал `TokenUsage`, обязательный возврат при ошибке апстрима, тесты на конвертацию.
- **Безопасность API-ключей.** Митигация: хранить только хеш, показывать plaintext один раз, IP-allowlist (ФАЗА 6), rate-limit, аудит-лог.
- **Зависимость от единственного апстрима (laozhang.ai).** Митигация: смарт-роутинг и фолбэк (ФАЗА 6), мониторинг SLA, абстракция провайдера в `api/services`.
- **Сложность two-service деплоя (Django + Next.js за Nginx).** Митигация: чёткая маршрутизация в `nginx.conf` (`/api`,`/admin`,`/media`,`/static` → Django, остальное → Next.js), Next.js standalone-сборка, отдельный `Dockerfile.frontend`.
- **Юридические требования B2B.** Митигация: оферта и реквизиты в ФАЗЕ 0 (до первых счетов), закрывающие документы (акт/УПД) в ФАЗЕ 4.
- **Дрейф контракта фронт↔бэк.** Митигация: типы генерируются из OpenAPI (`openapi-typescript`), версионирование `/api/v1/`, CI-проверка генерации типов.
- **Возврат эмодзи в интерфейс.** Митигация: CI-скрипт (Python/шаблоны) + ESLint-правило (TS/TSX) — регрессия падает на проверке.
