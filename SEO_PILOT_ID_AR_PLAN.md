## Риск, который нужно держать в голове

Google прямо предупреждает про **scaled content abuse** — публикация большого объёма AI-контента без реальной проверки может понизить ранжирование всего домена, а не только этих статей. Именно поэтому шаг 1 (нативная вычитка) — не формальность, а единственный реальный барьер перед тем, как это увидят поисковики.

## Исходный технический план (как согласовывали изначально)

### Контекст

Цель — начать пилот SEO-контента для id (Индонезия) и ar (MENA), качество и стратегия должны быть на уровне конкуренции с топ-1, с учётом актуальных SEO-практик 2026 года. Разведка подтвердила:

- Контент-машины для SEO-статей в кодовой базе не было — `create_seo_posts.py` пишет одну хардкод-статью, не LLM-пайплайн. Построили с нуля.
- `blog.Post`/`blog.Category` не имели поля языка вообще — ни в модели, ни в API, ни во фронтенде. Это противоречило GLOBAL_EXPANSION_PLAN.md §4.3, где для блога явно предписан **не** modeltranslation-подход — блог должен получить отдельное поле `language` и отдельные строки `Post` на каждый язык, так как статьи должны быть самостоятельным контентом под локальный рынок, а не переводом одного и того же русского текста.
- SEO/AI Overview 2026: 97% цитирований в AI Overviews берутся из страниц, уже попадающих в топ-20 органики — нельзя просто "писать под AI", нужны рабочие основы (E-E-A-T, структура, freshness, internal linking). Прямая политика Google против scaled content abuse.
- Три обязательных корректива (по итогам консультации с advisor):
  1. **Публиковать нельзя** без нативной вычитки — ни разработчик, ни пользователь не читают id/ar свободно.
  2. **Сначала пилот 2-3 статьи на язык**, потом остальное — проверить рендер до генерации всего батча.
  3. **Осторожность с темами для ar** — не давать материалам рамку "обход блокировки/санкций"; фокус на оплате без карты, качестве языка, кейсах для бизнеса/студентов.

### 1. Модель данных (`src/blog/models.py`)

- `Post.language` — CharField, choices ru/en/fa/tr/id/ar, default='ru', db_index=True.
- `Post.faq_items` — JSONField, список `{"question": str, "answer": str}` для FAQPage-схемы.
- `Category` — язык не добавляли (статьи пилота без категории).
- Миграция `0002_post_faq_items_post_language.py`.

### 2. API

- `BlogPostListView.get_queryset()` — фильтр по `?lang=`, при отсутствии — прежнее поведение (обратная совместимость для aineron.ru).
- Сериализаторы — добавлены поля `language`/`faq_items`.

### 3. Фронтенд

- `serverListBlogPosts`/`serverGetBlogPost` — добавлен `lang?: string`.
- Страницы блога передают `locale`.
- FAQPage JSON-LD добавлена рядом с Article/BreadcrumbList.
- RTL для ar — уже работал на уровне layout (`RTL_LOCALES`), ничего дополнительно строить не пришлось.

### 4. Контент-стратегия

12 брифов (6 id + 6 ar), каждый: `slug`, `title_hint`, `keywords`, `search_intent`, `angle`, `must_include_facts`, `related_model_hints`.

**id** — доступ к премиум-AI без карты/VPN, сравнение моделей, практика (бизнес/учёба/контент).
**ar** — оплата без карты, сравнение с ChatGPT Plus, кейсы бизнес/студенты. Без рамки "обход блокировки".

### 5. Пайплайн генерации

`src/blog/management/commands/generate_seo_articles.py`, модель `claude-sonnet-5` через laozhang. На статью — два вызова: draft (полный текст+FAQ) и self-critique (редактор ищет клише/галлюцинации/неестественность). Формат ответа — текстовые секции с разделителями `===MARKER===`, НЕ JSON (JSON ломался на длинной прозе). `is_published=False` жёстко в коде. Идемпотентно по slug. `--pilot` — первые 2-3 брифа на язык.

### Верификация (выполнено)

- `tsc --noEmit` на фронтенде — чисто.
- Django-миграция применена и на локальной тестовой БД, и на сервере (`showmigrations` подтвердил).
- Пилот (4 статьи) → структурная проверка (H2/H3, FAQ, кириллица, привязанные модели) → полный батч (12 статей) — все прошли.
- Рендер-логика (FAQPage JSON-LD, `network_slugs`, `language` в сериализаторе) проверена напрямую через DRF-сериализатор и код фронтенда — не через живой рендер страницы, так как флаг `is_published` намеренно не трогали даже для теста.
