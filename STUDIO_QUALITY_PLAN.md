# Studio — план качественной реализации (без обрезки)

> Анализ проведён Claude Opus 4.8. Цель — стать топ-1 по качеству vibe-coding в России.

---

## 0. TL;DR — главные открытия

Проблема **не в `max_tokens`**. В `coder.py` уже стоит `max_tokens=24000`, а файл на 416 строк обрезается на 130-й. Это значит **потолок зажат выше по стеку**, либо в LLM-прокси laozhang.ai.

Но **реальный главный баг** найден в нашем коде:

**`guardian.py:97` режет каждый файл до `content[:3000]` (~130 строк CSS)**. Guardian физически не видит файл дальше 130 строки → считает любой большой полный файл «обрезанным» → вечный fix-цикл. Симптом «обрыв на 130-й строке» — это ровно 3000 символов CSS.

## 1. Корневые причины всех проблем

### 1.1 Guardian слепой после 3000 символов (КРИТИЧНО — главный баг)

`guardian.py:97`:
```python
f'### {path}\n```\n{content[:3000]}\n```'
```

**Следствие**: Guardian никогда не видит файл дальше ~130 строк. Любой полный `globals.css` на 416 строк Guardian называет «обрезанным». Fix-итерация перегенерирует файл → Guardian снова видит 3000 символов → снова «обрезан» → бесконечный цикл.

### 1.2 `_repair_file` клеит скобки вместо полного восстановления

`coder.py:_repair_file` просит «только закрывающие скобки» (`max_tokens=4000`), показывая `content[-800:]`. Файл, обрезанный на 130/416 строк, нуждается в **280 строках**, не в скобках. Модель клеит `}` → `_is_truncated` проходит (файл кончается на `}`) → файл без 280 строк сохраняется как «полный» → Guardian снова видит неполный файл → бесконечный цикл.

### 1.3 Никто не читает фактическое потребление токенов

`base.py:run_prompt` не запрашивает `stream_options={"include_usage": True}` и не читает `chunk.usage`. Система не знает, сколько токенов реально вернула модель. `finish_reason` может приходить `stop` даже когда прокси обрезал ответ — и вся защита по `finish_reason == 'length'` не срабатывает.

### 1.4 JSON-упаковка кода — антипаттерн

`_run_legacy` просит `{"files": {"path": "полное содержимое"}}`. Три проблемы:
1. **Escaping**: каждый `"`, `\`, перевод строки экранируется → файл раздувается на 10–20%
2. **N файлов делят один бюджет**: 4 файла × 3000 токенов = 12000 токенов в одном вызове
3. **Один сломанный `\` рушит весь JSON** → теряются ВСЕ файлы шага

### 1.5 Fix-итерация перегенерирует весь файл вместо точечных правок

Когда Guardian говорит "добавь nav-ссылки", coder перегенерирует весь файл (400+ строк). Это снова упирается в тот же потолок. Нужно менять **только то что сломано**.

### 1.6 Нет continuation при достижении потолка

При `finish_reason == 'length'` coder только логирует и возвращает обрезанный stub. Нет автодозапроса «продолжи с места обрыва».

### 1.7 Architect проектирует монолиты

`globals.css` на 416 строк, `layout.tsx` на 362 строки — файлы-монолиты. Чем больше файл, тем выше риск упереться в потолок.

## 2. Как работают топовые сервисы

### Bolt.new — сырые блоки вместо JSON, запись по стриму

Bolt НЕ использует JSON для кода. Формат:

```
<boltArtifact id="..." title="...">
  <boltAction type="file" filePath="src/App.tsx">
...СЫРОЕ содержимое без экранирования...
  </boltAction>
  <boltAction type="shell">npm install</boltAction>
</boltArtifact>
```

Ключевые свойства:
- **Сырой текст между тегами** — никакого JSON-escaping
- **Запись по мере стрима** — файл материализуется прямо во время генерации
- **Один файл = один блок** — обрыв одного не рушит остальные

### Lovable / Cursor / Aider — diff-редактирование для изменений

Для fix-итераций топовые инструменты **не переписывают файл целиком**. Только изменяемый фрагмент:

```
src/App.tsx
<<<<<<< SEARCH
  const [count, setCount] = useState(0)
=======
  const [count, setCount] = useState(10)
>>>>>>> REPLACE
```

Чтобы поправить 1 строку в файле на 400 строк — модель выводит 6 строк, не 400. Именно поэтому у них нет обрезки при fix-итерациях.

### Continuation — дозапрос при достижении потолка

При `finish_reason == 'length'` инструменты подают уже сгенерированное как `assistant`-сообщение и просят «продолжи ровно с места обрыва». Повторяют до победного. Это не «допиши скобки» — полноценное дописывание оставшихся строк.

### Реалистичная декомпозиция на этапе архитектуры

v0/Replit/Bolt планируют так, чтобы файлы были < 250 строк: компоненты разбиты, стили модульные. Меньше монолитов → меньше шансов упереться в потолок.

## 3. Архитектурный план исправлений

### Приоритет 1 — Немедленные фиксы (делаем сейчас)

#### A. Guardian: убрать `content[:3000]`, добавить структурную проверку в коде

**Было** (`guardian.py:97`):
```python
f'### {path}\n```\n{content[:3000]}\n```'
```

**Стало**:
```python
f'### {path}\n```\n{content[:8000]}\n```'
```

И в prompts Guardian добавить:
```
Структурная целостность файлов (баланс скобок, обрезка) уже проверена
программно до тебя. НЕ выноси вердикт fix с обоснованием "файл обрезан".
Оценивай только: соответствие шагу, импорты, зависимости, логику.
```

**Добавить в tasks.py** перед отправкой в Guardian:
```python
def _structural_ok(path: str, content: str) -> bool:
    s = content.strip()
    if not s:
        return False
    if path.endswith(('.ts', '.tsx', '.js', '.jsx', '.css')):
        if abs(s.count('{') - s.count('}')) > 3:
            return False
    return s[-1] in ('}', ';', '>', ')', '"', "'")
```

Файлы не прошедшие `_structural_ok` → сразу в continuation coder, не в Guardian как «обрезанные».

#### B. base.py: читать фактические токены вывода

```python
# Добавить в run_prompt:
stream_options={'include_usage': True}

# Читать из чанков:
if getattr(chunk, 'usage', None):
    self.last_completion_tokens = chunk.usage.completion_tokens or 0
```

Это позволит видеть: если `completion_tokens` стабильно == 4096 на каждом обрезанном файле → потолок прокси подтверждён.

#### C. coder.py: continuation вместо `_repair_file`

```python
def _continue_output(self, system: str, user: str, prior: str, model: str,
                     max_tokens: int, max_rounds: int = 5) -> str:
    """Дозапрашивает модель пока finish_reason == 'length'."""
    full = prior
    for attempt in range(max_rounds):
        messages = [
            {'role': 'system', 'content': system},
            {'role': 'user', 'content': user},
            {'role': 'assistant', 'content': full},  # подаём уже сгенерированное
        ]
        part = self._raw_prompt(messages, model=model, max_tokens=max_tokens)
        full += part
        capped = (
            self.last_finish_reason == 'length'
            or (self.last_completion_tokens
                and self.last_completion_tokens >= max_tokens - 64)
        )
        if not capped:
            break
        self.log(f'Continuation {attempt + 1}: дозапрашиваю (токены={self.last_completion_tokens})', level='warning')
    return full
```

**Удалить `_repair_file`** — заменён continuation.

### Приоритет 2 — Блочный формат вместо JSON (следующий этап)

Заменить `_run_legacy` (`{"files": {...}}`) на сырые блоки:

**Новый промпт кодера:**
```
Выводи файлы строго в формате:

===FILE: relative/path/file.tsx===
<сырое содержимое файла, без экранирования, без markdown>
===ENDFILE===

Каждый файл ОБЯЗАТЕЛЬНО закрыт строкой ===ENDFILE===.
Если файл крупный — выводи целиком, тебя продолжат автоматически.
```

**Парсер:**
```python
FILE_RE = re.compile(r'===FILE:\s*(.+?)\s*===\n(.*?)\n===ENDFILE===', re.DOTALL)

def _parse_file_blocks(self, text: str) -> dict:
    files = {}
    for m in FILE_RE.finditer(text):
        path, content = m.group(1).strip(), _strip_fences(m.group(2))
        if path and content:
            files[path] = content
    return files
```

**Преимущества над JSON:**
- Нет escaping — кавычки и бэкслеши идут как есть
- Обрыв последнего блока не рушит уже завершённые файлы
- Continuation естественно достраивает незакрытый блок

**Детектор незакрытого блока** для continuation:
```python
unclosed = text.count('===FILE:') > text.count('===ENDFILE===')
```

### Приоритет 3 — Search/Replace для fix-итераций

Для fix-режима coder выдаёт точечные правки:

**Промпт fix-режима:**
```
Внеси МИНИМАЛЬНЫЕ правки. НЕ переписывай файлы целиком.
===EDIT: path/file.tsx===
<<<<<<< SEARCH
<дословный текущий фрагмент>
=======
<новый фрагмент>
>>>>>>> REPLACE
===ENDEDIT===
```

**Применение:**
```python
def _apply_edits(self, raw: str, existing_files: dict) -> dict:
    results = {}
    for m in EDIT_RE.finditer(raw):
        path, search, replace = m.group(1), m.group(2), m.group(3)
        cur = existing_files.get(path, '')
        if search.strip() in cur:
            results[path] = cur.replace(search.strip(), replace.strip(), 1)
        else:
            # SEARCH не совпал → полная перегенерация только этого файла
            results[path] = None  # сигнал на per-file rewrite
    return results
```

Это полностью устраняет «fix-итерация генерирует весь файл 400 строк и снова обрезается».

## 5. Промпт-инжиниринг

### 5.1 Coder — блочный формат

```
Ты senior-разработчик. Реализуй РОВНО ОДИН шаг.

ФОРМАТ ВЫВОДА (строго, ничего вне блоков):
===FILE: relative/path===
<полное сырое содержимое файла — без markdown ```, без экранирования>
===ENDFILE===

ЖЁСТКИЕ ПРАВИЛА:
- Содержимое файла — СЫРОЙ текст. НЕ экранируй кавычки и слеши.
- Каждый файл ОБЯЗАТЕЛЬНО закрыт строкой ===ENDFILE===.
- Если файл крупный — всё равно выводи целиком; система дозапросит автоматически.

```

```
Продолжай вывод РОВНО с места обрыва. Не повторяй уже выведенное.

## 6. Как выглядит правильная архитектура после всех изменений

```
Пользователь описывает проект
         ↓
    Architect (claude-opus-4-8)
    → PROJECT.md (≤1000 слов)
    → COMMITS.md (шаги с декомпозицией, файлы ≤250 строк каждый)
         ↓
    Для каждого шага:
         ↓
    Coder (claude-sonnet-4-6)
    → Формат ===FILE===/===ENDFILE=== (не JSON)
    → При finish_reason='length' → continuation
    → Запись каждого файла независимо по мере парсинга
         ↓
    _structural_ok() — детерминированная проверка (код, не LLM)
    → Если не ок → continuation в coder, не в Guardian
         ↓
    Guardian (claude-sonnet-4-6)
    → Видит файлы целиком ([:8000])
    → Только логика/импорты/стек, НЕ "обрезка"
    → PASS: следующий шаг
    → FIX: search/replace блоки, не перезапись
         ↓
    Sandbox: build check с реальными ошибками
```

## 7. Метрики успеха

- **0 итераций** по причине «файл обрезан» (Guardian не видит обрезку)
- **< 2 итераций** на шаг в среднем (Guardian только по логике)
- **Файлы 400+ строк** генерируются полностью (continuation)
- **Fix-итерации** меняют 5–20 строк, не 400
- **Архитектор** проектирует файлы ≤ 250 строк
