## 0. Зафиксированные решения

| Решение | Значение |
|---------|----------|
| Домен | Отдельный международный домен (aineron.com / .ai / .app — купить) |
| Инфраструктура | **Второй сервер вне РФ**, полностью отдельный инстанс (код общий, деплой и БД свои) |
| Оплата | **Только криптовалюта** на старте (USDT/TON), позже — Telegram Stars и локальные PSP |
| Валюта продукта | **Кредиты** (внутренне = те же kopecks, 1 кредит = 1 внутренняя единица) |
| Языки | **Английский — базовый и обязательный** + языки целевых рынков (см. §2). Русского на международной версии НЕТ — Россию и СНГ обслуживает aineron.ru |
| Исключённые рынки | Франция, Германия, ЕС в целом, США, Великобритания, Китай — «жирные» конкурентные рынки без нашего преимущества |
| Под вопросом (волна 2) | Индия, Пакистан/Бангладеш, Вьетнам — решение после метрик волны 1 |

Логика выбора рынков: идём туда, где есть **гео-барьер** (AI-сервисы заблокированы) или **платёжный барьер** (местной картой нельзя оплатить ChatGPT), слабый английский и слабая конкуренция. Это в точности та формула, на которой выросли в России.

## 2. Архитектура: два независимых инстанса

```
aineron.ru  (сервер РФ, как сейчас)          aineron.com (новый сервер, Hetzner/аналог, ЕС)
├── рубли, Robokassa + Stars + крипта        ├── кредиты, только крипта
├── язык: ru                                 ├── языки: en, fa, tr, id, ar
├── своя БД                                  ├── СВОЯ БД (пользователи не пересекаются)
└── деплой как сейчас                        └── тот же репозиторий, свой .env + compose
```

Принципы:

- **Один репозиторий, два деплоя.** Никакого форка кода — международное поведение включается env-флагами. Иначе через три месяца будет два расходящихся проекта.
- Новые env-переменные:
  ```
  INTL_MODE=1                  # международный инстанс
  DEFAULT_LOCALE=en
  ENABLED_LOCALES=en,fa,tr,id,ar
  CURRENCY_DISPLAY=credits     # credits | rub
  PAYMENTS_ENABLED=crypto      # список каналов: crypto | robokassa | stars
  ```
- `INTL_MODE=1` отключает: Robokassa, юр-страницы РФ, русскоязычные management-команды контента; включает: крипто-чекаут, кредитную витрину, мультиязычный роутинг.
- Сервер: вне РФ (Hetzner / Contabo / OVH). Причины: доступность для Ирана и Турции (российские IP-диапазоны там нестабильны), доверие аудитории, независимость от РФ-инфраструктуры. Провайдеры laozhang.ai / apimart.ai работают с любого сервера.
- Docker Compose тот же (web, celery, redis, db, frontend, nginx); Studio/Gitea-сервисы на международной версии на старте можно не поднимать (`STUDIO_ENABLED=0`) — меньше поверхности, меньше ресурсов.
- Юр-тексты: privacy policy / terms на английском, нейтральная юрисдикция; `LegalDocument` уже управляется из админки — просто другой контент.

## 3. Биллинг: кредиты + крипта

### 3.1. Кредиты

Внутри ничего не меняем — `balance_kopecks`, идемпотентный ledger, `spend_kopecks`/`add_kopecks` работают как есть. Меняется только **витрина**:

- 1 кредит = 1 внутренняя единица (бывшая копейка). Цены моделей уже лежат в `cost_kopecks` — на международной версии это просто «кредиты за сообщение».
- `frontend/lib/money.ts` → добавить `formatCredits(amount, locale)`; выбор `formatRub`/`formatCredits` по `CURRENCY_DISPLAY`.
- Пакеты пополнения в USD-номинале: например $5 / $10 / $25 / $50 → кредиты по фиксированному курсу пакета (не по биржевому курсу рубля — международная цена назначается заново, исходя из себестоимости токенов в $; это отдельная строка в TARIFFS-экономике).
- Слово «звёзды» на международной версии не используется нигде (конфликт с Telegram Stars).

### 3.2. Крипто-платежи (единственный канал на старте)

Двухступенчатая стратегия:

**Шаг 1 — Crypto Pay API (@CryptoBot).** Запуск за 1–2 дня:
- REST API: создать инвойс (USDT/TON/BTC/ETH) → получить ссылку → редирект/кнопка.
- Webhook подтверждения с подписью.
- Работает и на сайте (ссылка-инвойс), и в боте нативно.
- Комиссия ~1–3%; вывод в USDT/TON.

**Шаг 2 — прямой приём USDT TRC-20 / TON** (когда пойдёт объём): собственные депозит-адреса или memo-инвойсы, индексация сети, ноль посредников и комиссий. Отдельный мини-проект, не блокирует запуск.

Интеграция в наш биллинг (по образцу Robokassa):

```
src/users/crypto_payments.py
  create_crypto_invoice(user, usd_amount) → PaymentHistory(status='pending',
      payment_type='crypto', external_id=invoice_id)

POST /users/api/payment/crypto/callback/   (webhook)
  проверка подписи → user.add_kopecks(credits, type='topup',
      reference=f'crypto:{invoice_id}')      # идемпотентно из коробки
```

- Инвойс фиксирует сумму кредитов в момент создания, TTL 30 минут.
- Тот же модуль включаем и на aineron.ru (пополнение криптой для СНГ) — одна интеграция, два инстанса.
- UI: страница `/account/billing/` — блок «Pay with crypto» (USDT TRC-20 / TON / BTC), статус-поллинг инвойса, как сейчас у Robokassa-флоу.

## 4. Мультиязычность — как делать правильно

### 4.1. Frontend: next-intl

Стандарт де-факто для Next.js 14 App Router — **next-intl**. Что делается:

1. **Реструктуризация роутинга**: `frontend/app/*` → `frontend/app/[locale]/*` (лендинг, каталог, чат, кабинет, блог и т.д.). Технические файлы (`manifest.ts`, `robots.ts`, `api`-роуты) остаются снаружи.
2. **middleware.ts**: определение локали — путь → cookie `NEXT_LOCALE` → заголовок `Accept-Language` → `en`. Редирект `/` → `/en/` (или serve default без префикса — выбрать `localePrefix: 'always'`, для hreflang чище).
3. **Словари**: `frontend/messages/en.json`, `fa.json`, `tr.json`, `id.json`, `ar.json` — формат ICU Message Format (плюрали, подстановки):
   ```json
   { "chat": { "insufficientBalance": "Not enough credits. You need {count, plural, one {# credit} other {# credits}}." } }
   ```
4. **Извлечение строк** — самая трудоёмкая часть: все хардкод-строки в `app/` и `components/` заменить на `t('key')`. Делать постранично, начиная с денежного пути: лендинг → регистрация → каталог → чат → биллинг. Строки глубоких настроек можно переводить второй итерацией.
5. **generateMetadata** per-locale (title/description из словаря), `sitemap.ts` с `alternates.languages` (hreflang), JSON-LD на языке страницы.
6. **CI-проверка**: скрипт, падающий при недостающих ключах в любой локали (сравнение ключей с en.json) — добавить рядом с `check_no_emoji.py`.

### 4.2. RTL и шрифты (критично для fa и ar)

- `<html lang={locale} dir={locale === 'fa' || locale === 'ar' ? 'rtl' : 'ltr'}>` в корневом layout.
- Tailwind: перейти на **logical properties** — `ms-*/me-*` вместо `ml-*/mr-*`, `ps-*/pe-*` вместо `pl-*/pr-*`, `text-start/text-end` вместо `text-left/right`. Tailwind сам разворачивает их в RTL. Пройтись поиском по `ml-|mr-|pl-|pr-|left-|right-|text-left|text-right` и заменить там, где это направление текста, а не абсолютное позиционирование.
- Шрифты: латиница/кириллица — Inter (как сейчас); **fa — Vazirmatn**; **ar — IBM Plex Sans Arabic или Noto Naskh Arabic**. Подключить через `next/font` с `subsets`, выбирать по локали в layout.
- Компоненты со «направленной» логикой (карусели, стрелки «назад», чат-пузыри) — проверить вручную на fa.

### 4.3. Backend (Django/DRF)

- **Ошибки и сообщения API → коды, не текст.** `{"error": {"code": "insufficient_balance", "params": {"required": 500}}}` — текст рендерит фронт из словаря. Это надёжнее и дешевле, чем gettext по всему DRF, и не требует передачи локали в Celery-задачи.
- **Переводимый контент БД** — `django-modeltranslation` (колонки `field_en`, `field_fa`, ...):
  - `NeuralNetwork`: `description`, `seo_title`, `seo_description`
  - `FAQ`: `question`, `answer`
  - `Tariff`/пакеты: `name`, `description`
  - `Category`: `name`
  - API отдаёт язык по `?lang=` или `Accept-Language` (fallback: en).
- **Блог**: НЕ переводить посты через modeltranslation. SEO-статьи на каждом языке — это отдельные статьи под локальные ключевики (см. §4.5). Добавить `Post.language` + фильтр в API.
- Транзакционные email: шаблоны per-locale, у пользователя поле `language` (заполняется при регистрации из локали фронта).

### 4.4. Чем переводить — пайплайн

Целевые языки (fa, ur, bn — вне покрытия DeepL; ar/tr/id DeepL умеет, но без контекста продукта), поэтому:

**Основной инструмент — LLM-перевод через собственный доступ (laozhang.ai), моделью уровня Claude Sonnet / GPT-4o.** Преимущества: покрывает все наши языки, понимает контекст («credits» ≠ «титры»), сохраняет ICU-плейсхолдеры, стоит копейки на нашем же канале.

Пайплайн (оформить management-командой `translate_locales`):

1. **Источник истины — en.json** (написать английскую версию вручную/с вычиткой — это база, с которой переводится всё).
2. **Глоссарий** `frontend/messages/_glossary.json`: фиксированные термины — credits, top up, prompt, model, названия продуктов (не переводятся), tone of voice. Подаётся в system prompt переводчика.
3. Скрипт: для каждого ключа, отсутствующего в `{locale}.json`, — батч-перевод LLM с контекстом (`ключ`, `en-строка`, `глоссарий`, `описание экрана`). Инкрементально: переводятся только новые/изменённые ключи (хранить hash en-строки).
4. **Валидация автоматом**: плейсхолдеры `{count}` сохранены, ICU-синтаксис парсится, длина строки не взорвала кнопку (soft-лимит ×1.6 от en).
5. **Вычитка носителем — обязательна для денежного пути** (лендинг, прайсинг, чекаут, онбординг) на fa/tr/id/ar. Заказывается на Upwork/локальных биржах, объём небольшой (~200–400 строк). Остальной интерфейс идёт машинным переводом и правится по жалобам.
6. Тот же пайплайн — для контента БД (`translate_catalog`): описания моделей, FAQ.

Что НЕ делать: не подключать Crowdin/Lokalise на старте (оверхед для соло-команды; JSON в git + LLM-пайплайн покрывает всё), не использовать Google Translate API (качество маркетинговых строк заметно хуже LLM).

### 4.5. SEO на локальных языках

Главный канал органики — на fa/tr/id/ar контентная конкуренция в AI-нише в разы ниже, чем в en:

- Расширить существующую контент-машину (`create_seo_posts`) параметром языка: локальные ключевики («ChatGPT بدون فیلتر», «yapay zeka türkçe», «AI tanpa VPN») → 100–200 программных статей на язык.
- hreflang-связки между локалями + x-default=en.
- Страницы моделей `/[locale]/models/[slug]/` с переведёнными описаниями и JSON-LD.
- Иран: Google работает через фильтр, но им пользуются; плюс локальные каналы дистрибуции — Telegram-каналы важнее SEO.

### 4.6. Telegram-бот международной версии

- **Отдельный бот** (@aineron_global_bot или per-market боты) на международном инстансе — не смешивать с русским.
- Локаль: `from_user.language_code` + выбор в /settings, поле `TelegramUser.language`.
- Строки хендлеров → словарь `telegram_bot/locales/{en,fa,tr,id,ar}.json` + хелпер `t(user, key)` (aiogram 3 позволяет и `aiogram.utils.i18n` с gettext, но JSON-словарь единообразнее с фронтовым пайплайном — переводится той же командой).
- Оплата в боте: инвойсы Crypto Pay (нативно в Telegram) — для Ирана бот, вероятно, станет основным интерфейсом, важнее сайта.

## 6. Позиционирование по рынкам

«Без VPN» не переводится дословно — на каждом рынке своя боль:

| Рынок | Сообщение |
