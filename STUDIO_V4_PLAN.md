## 0. Что уже работает и НЕ переделывается в V4

Прежде чем планировать новое, фиксируем реальность кода — иначе план теряет доверие команды на первой странице. Несколько «ограничений» из постановки **уже частично решены**, и V4 не должен их «решать» заново. Ниже честная таблица: что есть, и где настоящий открытый зазор.

| Заявленное ограничение | Что РЕАЛЬНО есть в коде | Настоящий открытый зазор (его и закрываем) |
|------------------------|--------------------------|---------------------------------------------|
| «Нет per-step проверки звёзд, юзер уходит в минус» | `tasks.py:_billing_charge` списывает звёзды на каждом шаге; `InsufficientStars` + `_pause_no_funds` ставят паузу `paused_on_loop`; `billing.py:reserve`/`charge_from_reserve` резервируют под оценку; `max_stars_budget` — потолок проекта | Биллинг берёт **фиксированный `AGENT_BUDGET`-констант**, а не реальные токены. `base.py:92` читает только `completion_tokens`, **prompt/input-токены не считаются вообще**. `estimate_stars` использует жёсткий `planned_steps=5` — резерв недо-оценивает длинные проекты |
| «Runtime error → fix loop не подключён» | `views/pipeline.py:ConsoleErrorView` (305-316) уже пишет `fix_plan` и дёргает `coder_iteration.delay()` | Нет авто-захвата ошибок (всё держится на POST с фронта), нет повторного build-check и Guardian-ревью **после** фикса, нет dedup/лимита итераций фикс-петли |
| Один провайдер (laozhang.ai) | `base.py:get_client()` — единственный `OpenAI(base_url=…)` клиент | Нет fallback при 5xx/таймауте провайдера → весь pipeline падает |
| Стрим файла в UI | `events.py:publish_event` (Redis pub/sub), `base.py:run_prompt` уже стримит дельты внутри себя | Дельты **не транслируются** во фронт — файл появляется только целиком после генерации |
| Guardian-контекст | `guardian.py` видит `content[:8000]` изменённых файлов | Не видит остального проекта → ломает кросс-файловые контракты (импорты, пропсы) |

**Не трогаем (работает):** FILE_BLOCKS-парсер (`blocks.py`), EDIT-блоки (`edits.py`), структурные/dep-валидаторы (`validators.py`), continuation по маркеру (`base.py:run_prompt_with_continuation`), sandbox + preview-прокси, Gitea-репозитории, same-diff loop detection, `ESCALATION_MAP`, ветвление/коллабораторы/шаблоны. V4 строится **поверх**.

## 1. Стратегическое позиционирование

### Почему мы не догоняем bolt.new по UX-паритету в лоб

bolt.new и StackBlitz построены на **WebContainers** — проприетарной технологии запуска Node.js целиком в браузере (WASM). Это их моат, и он не воспроизводим: мы используем Docker-sandbox на сервере. Гнаться за «мгновенным HMR в браузере» — это сжигать кварталы на чужой технологический ров. **Мы этого делать не будем.** Вместо этого V4 закрывает *воспринимаемый* разрыв двумя дешёвыми приёмами (стрим написания файлов + надёжная петля исправления рантайм-ошибок) — этого достаточно, чтобы пользователь чувствовал тот же «живой» поток.

### Где наш собственный, невоспроизводимый моат

Глобальные конкуренты (bolt, Lovable, v0) **никогда** не будут приоритизировать российскую локализацию: для них это <1% рынка с регуляторными рисками. Это и есть наш ров:

1. **Российский стек по умолчанию** — сгенерированное приложение сразу умеет Робокасса, VK ID, Яндекс.Карты, Telegram Login, и деплоится на Selectel/Timeweb (Vercel в РФ под VPN — мёртв при рождении для конечного пользователя в России).
2. **Telegram Mini App Studio** — генерация и деплой настоящих Telegram Mini Apps с российскими платежами. **Нулевая конкуренция в мире и в России.** Telegram — доминирующая платформа в РФ/СНГ; «навайбкодить мини-апп с оплатой за 10 минут» не умеет никто.

Позиционирование V4 в одной фразе: **«Bolt для российского продукта: тот же поток генерации, но на выходе — приложение, которое реально работает и продаётся в России, вплоть до Telegram Mini App с оплатой».**

## 2. Фазы реализации

Четыре независимо деплоимых фазы. Порядок не произволен: каждая снимает блокер для следующей.

### Фаза 0 — Надёжность и стоимость (2 недели)

**Цель:** перестать терять маржу на невидимых input-токенах и перестать падать, когда провайдер моргнул. Починить рантайм-петлю «по-настоящему».

**Почему первой:** недавнее изменение (полный COMMITS.md в каждый промпт) раздуло именно **input-токены**, которые сейчас не считаются ни в биллинге, ни в метриках — это прямая утечка маржи. Чинить UX поверх убыточной экономики бессмысленно. Фаза разблокирует доверие к цифрам перед тем, как мы начнём добавлять более дорогие фичи (Фазы 2-3).

**Что разблокирует:** честный per-step учёт → корректный `estimate_stars` → корректный резерв → можно агрессивно эскалировать на opus в сложных шагах, зная реальную цену.

#### Архитектурные решения

1. **Учёт реальных токенов (`base.py`).** В `run_prompt` при `chunk.usage` сохранять и `prompt_tokens`, и `completion_tokens` → новые поля `last_prompt_tokens`, `last_total_tokens`. Сейчас читается только `completion_tokens` (строка 92) — input невидим.
2. **Биллинг по факту, а не по константе (`billing.py`, `tasks.py`).** `_billing_charge` будет считать `cost = ceil((prompt + completion)/1000 * STAR_RATE[tier])` из реальных `last_*_tokens` агента вместо фиксированного `AGENT_BUDGET`. `AGENT_BUDGET` остаётся **только** для `estimate_stars`/резерва (предоплата). За флагом `STUDIO_V4_TOKEN_BILLING`.
3. **Контекст-кэш COMMITS.md (`tasks.py`, агенты).** Вместо пересылки полного `commits_md_content` в каждый промпт coder/guardian — передавать только текущий шаг (`_get_step_text` уже умеет резать) + краткое summary остальных шагов. Это снимает корень роста input-токенов.
4. **Динамический `estimate_stars` (`billing.py`).** Заменить `planned_steps=5` на реальное число шагов из `interview_data['plan']` (после Architect оно известно).
5. **Provider fallback (`base.py:get_client`).** Обернуть `chat.completions.create` в try/except на `APIError/Timeout/5xx` → ретрай на вторичный клиент (`LAOZHANG_API_URL_FALLBACK`, env). Логировать `provider_failover` метрикой.
6. **«По-настоящему» рантайм-петля (`views/pipeline.py:ConsoleErrorView`, `tasks.py`).** Петля уже дёргает `coder_iteration` — добавить недостающее: (а) после фикса принудительный `run_build_check` + Guardian-ревью изменённого файла; (б) dedup ошибок по хэшу `message+file` (как `same-diff` в `tasks.py:344`); (в) лимит `max_autofix=3`, после — пауза с человеко-читаемым reason; (г) клиентский хук в preview-iframe, который ловит `window.onerror`/`console.error` и шлёт на эндпоинт (сейчас полагается на ручной POST).

#### Риски и митигации

| Риск | Митигация |
|------|-----------|
| Биллинг по факту → юзер видит «плавающую» цену, путается | UI: показываем оценку (резерв) + финальный факт; резерв всегда ≥ факта, излишек возвращается `release_reserve` |
| Fallback-провайдер даёт другой стиль кода | Fallback только при ошибке primary, не A/B; модель та же (laozhang проксирует те же id) |
| Автофикс-петля зацикливается | dedup по хэшу + `max_autofix=3` + пауза (паттерн уже есть для same-diff) |

#### Метрики успеха

- **Margin leak closed:** разница между фактически списанным и реально потраченным (по usage) < 5% (сейчас неизмеримо).
- **Input-токены** логируются per-step в `interview_data['metrics']`.
- **Provider availability:** доля шагов, спасённых fallback-ом, > 0 при инцидентах; pipeline-failure из-за провайдера → 0.
- **Autofix resolution rate:** доля рантайм-ошибок, исправленных ≤3 итерациями, > 70%.

### Фаза 1 — UX-паритет с bolt.new (4 недели)

**Цель:** закрыть *воспринимаемый* разрыв с bolt — «смотреть, как пишется код» + надёжное живое превью + Guardian, который видит весь проект.

**Почему после Фазы 0:** стрим и более умный Guardian увеличивают нагрузку на провайдера и токены — без честного учёта (Ф0) мы не увидим их цену.

**Что разблокирует:** ощущение «живого» инструмента — ключевой фактор удержания на демо/онбординге, и фундамент, на котором показ генерации TMA (Фаза 3) будет выглядеть premium.

#### Архитектурные решения

1. **Стрим написания файла во фронт (`coder.py`, `events.py`, `frontend/components/studio/CodeViewer.tsx`).** `base.run_prompt` уже стримит дельты внутри себя — добавим необязательный коллбэк `on_delta`. `coder._generate_one_file` будет передавать коллбэк, который батчит дельты (~каждые 200 мс / 80 символов) и шлёт `publish_event(..., {'type': 'file_delta', 'path': ..., 'chunk': ...})`. Фронт (`CodeViewer`/`AgentLog`) подписан на тот же SSE-канал (`PipelineEventsView`) — открывает файл и до-печатывает дельты. Технический выбор: **батчинг обязателен** — Redis pub/sub на каждый токен убьёт и Redis, и фронт. За флагом `STUDIO_V4_STREAMING`.
2. **Guardian с контекстом всего проекта (`guardian.py`).** Сейчас Guardian видит только изменённые файлы (`content[:8000]`). Добавим **карту проекта**: для каждого *неизменённого* файла — путь + экспортируемые символы (дешёвый regex по `export`), а полный текст — только для файлов, на которые ссылается текущий шаг. Это ловит кросс-файловые поломки (неверный импорт, изменившийся проп) без раздувания токенов на весь проект.
3. **Улучшение живого превью (`sandbox.py`, `views/pipeline.py:PreviewProxyView`).** HMR в sandbox отключён (`hmr:false` в scaffold) ради стабильности. Не включаем HMR (источник флака), вместо этого: после коммита шага — авто-`PreviewRestartView` + SSE-событие `preview_ready`, фронт перезагружает iframe мягко (без полного flash). Опционально — лёгкий dev-watch, перезапускающий только при изменении файлов шага.

#### Риски и митигации

| Риск | Митигация |
|------|-----------|
| Стрим дельт перегружает Redis/SSE | Батчинг по времени+размеру; флаг отключения; дельты не персистятся (только финальный `StudioFile`) |
| Карта проекта раздувает Guardian-токены | Только сигнатуры (`export`-строки), полный текст — лишь для зависимостей шага |
| Частые preview-restart раздражают | Debounce: рестарт раз в шаг (после commit), не на каждый файл |

#### Метрики успеха

- **Time-to-first-visible-code** (от старта шага до первого символа в UI) < 3 с.
- **Cross-file break rate** (поломки импортов/пропсов, пойманные после шага) падает > 50% vs V3.
- **Preview refresh latency** (commit → живое превью) < 8 с.

**Цель:** на выходе Studio даёт приложение, готовое к российскому рынку: Робокасса, VK ID, Яндекс.Карты, Telegram Login, деплой на Selectel/Timeweb, поддержка GigaChat.

**Почему после Фазы 1:** российские интеграции — это новые шаблоны и scaffold, требующие зрелого, стабильного pipeline (Ф0) и хорошего показа генерации (Ф1), чтобы premium-фича выглядела premium.

**Что разблокирует:** **прямой блокер Фазы 3** — Telegram Mini App требует публичного российского HTTPS-хостинга и российских платежей. Vercel в РФ под VPN → деплой TMA туда мёртв при рождении. Сначала российский деплой и платежи, потом — TMA поверх них.

#### Архитектурные решения

1. **Шаблоны с предзашитыми интеграциями (`scaffold.py`, `management/commands/seed_templates.py`).** Новые `StudioTemplate` + scaffold-словари:

   - **Telegram Login Widget**: компонент + серверная проверка подписи `hash`.
   Scaffold-словарь надёжнее, чем доверять LLM (паттерн уже применён в V3 для UI-примитивов).
2. **Деплой на Selectel/Timeweb (`tasks.py`).** Рядом с `deploy_to_vercel` — `deploy_to_timeweb` / `deploy_to_selectel`: пуш в Gitea-репо проекта → триггер сборки через API провайдера (Timeweb Apps / Selectel Container Registry + S3 для статики). `DeployView` принимает `target` (`vercel|github|timeweb|selectel`). За флагом `STUDIO_V4_RU_STACK`.
3. **GigaChat (`models_catalog.py`, `base.py`).** Добавить `{'id': 'gigachat-pro', 'tier': 'smart', …}` и `gigachat-lite` (`fast`) в `STUDIO_MODELS`/`MODEL_TIER`/`DEFAULT_AGENT_MODELS`. Если GigaChat недоступен через laozhang-прокси — отдельный клиент в `get_client` по префиксу id `gigachat-*` (Sber API, OAuth-токен). Это органично использует уже добавленный в Ф0 provider-fallback-механизм.

#### Риски и митигации

| Риск | Митигация |
|------|-----------|
| Секреты платёжек в сгенерированном коде | Только `.env.example` + код, читающий `process.env`; реальные ключи — никогда в репо |

- **GigaChat usage** > 0 (валидирует, что интеграция жива).

### Фаза 3 — Суперфича: Telegram Mini App Studio (8 недель)

**Цель и заголовок V4.** Пользователь описывает идею → Studio генерирует **настоящий Telegram Mini App** (TWA) с российской оплатой → деплоит на российский хостинг → отдаёт готовую ссылку для BotFather. Мировая первая фича, нулевая конкуренция.

**Почему последней:** TMA — это композиция всего предыдущего: стабильный pipeline (Ф0), premium-показ генерации (Ф1), российский хостинг + Робокасса (Ф2). Раньше Ф2 её делать нельзя — деплоить некуда.

**Что разблокирует:** уникальное рыночное предложение, которого нет ни у одного глобального и российского конкурента.

#### Архитектурные решения (самая детальная часть)

1. **Новый target stack `"tma"` (`models.py`, `scaffold.py`).** Рядом с `nextjs/react/vue/html` — пятый стек `tma`. Технический выбор стека: **Vite + React + `@twa-dev/sdk`** (telegram-web-app SDK). Vite, потому что у нас уже отлажен Vite-sandbox (`server:{host:true,port:3000,hmr:false}`), а TMA — это SPA внутри Telegram WebView. Целевой стек влияет на: scaffold, промпты Architect/Coder, DESIGN.md, деплой.

2. **TMA-scaffold (`scaffold.py`).** Шаг 1 ставит готовый каркас:
   - `index.html` с `<script src="https://telegram.org/js/telegram-web-app.js">`.
   - `src/lib/telegram.ts` — обёртка над `window.Telegram.WebApp`: `ready()`, `expand()`, `MainButton`, `BackButton`, `HapticFeedback`, чтение `initDataUnsafe.user`.
   - `src/lib/theme.ts` — маппинг Telegram theme params (`themeParams.bg_color`, `text_color`, `button_color`) в CSS-переменные. **Ключевой UX-момент:** TMA обязан выглядеть нативно в теме пользователя Telegram — DESIGN.md для `tma` использует эти токены, а не свою палитру.
   - Серверная часть (на выбранном RU-хостинге из Ф2): **валидация `initData`** — проверка HMAC-SHA256 подписи `initData` секретным ключом бота (`hash` vs пересчитанный). Без этого любой может подделать `user`. Это **обязательный** scaffold-файл, не опциональный.

3. **Российские платежи в TMA (переиспользует Ф2).** Два пути, оба через `Telegram.WebApp`:
   - **Telegram Payments через Робокасса-провайдера**: бот вызывает `createInvoiceLink` с provider_token от Робокасса (выдаётся через BotFather → `/setpayments`). Scaffold даёт серверный роут, создающий invoice, и клиентский `WebApp.openInvoice(url, callback)`.
   - **Прямой Робокасса-редирект** (для подписок/сложных сценариев) — переиспользуем шаблон Робокасса из Ф2.

4. **Промпты Architect/Coder для TMA (`architect.py`, `coder.py`).** При `target_stack == 'tma'`:
   - Architect получает правила: «это Telegram Mini App, не сайт»; обязательны `telegram.ready()`/`expand()` при старте; навигация через `BackButton`, не через свой роутер-хедер; основное действие — через `MainButton`; тема — из `themeParams`.
   - Coder получает в системный промпт выжимку `src/lib/telegram.ts` API — обязан использовать обёртку, а не сырой `window.Telegram`.

5. **Деплой TMA (`tasks.py:deploy_to_timeweb`, новый `tma_publish`).** Переиспользует RU-деплой Ф2 (TMA = статический SPA + лёгкий серверный роут для invoice/initData-валидации). После деплоя — `tma_publish` отдаёт пользователю: (а) публичный HTTPS-URL мини-аппа; (б) пошаговую инструкцию для BotFather (`/newapp`, привязка URL); (в) опционально — авто-регистрацию через Bot API, если пользователь дал bot token.

6. **TMA-preview (`views/pipeline.py:PreviewProxyView`, frontend `PreviewPanel.tsx`).** Превью оборачивается в **эмулятор Telegram-окружения**: фронт инжектит mock `window.Telegram.WebApp` (фиктивные `initData`, `themeParams`, заглушки `MainButton`/`HapticFeedback`) — чтобы мини-апп рендерился в обычном iframe вне Telegram. Технический выбор: mock-SDK на стороне фронта, а не запуск настоящего Telegram — настоящий доступен только после деплоя по HTTPS-ссылке.

#### Риски и митигации

| Риск | Митигация |
|------|-----------|
| Невалидная подпись `initData` → дыра в безопасности | Валидация HMAC — обязательный scaffold-файл; Guardian-чек «есть ли проверка initData» для tma-стека |
| Робокасса provider_token требует ручной настройки в BotFather | `tma_publish` даёт точную пошаговую инструкцию; не пытаемся автоматизировать BotFather-чат |
| Превью без настоящего Telegram вводит в заблуждение | Mock-SDK + явная плашка «Превью — эмуляция Telegram; реальная тема и оплата — после деплоя» |
| Telegram меняет WebApp API | Версионируем `telegram.ts`-обёртку, обновляем scaffold централизованно |

#### Метрики успеха

- **TMA generation success rate** (дошёл до `completed` + собрался) > 80%.
- **TMA deploy + BotFather-ready rate** > 70%.
- **Payment-wired TMA share** — доля TMA-проектов с рабочим invoice-флоу.
- **Headline KPI:** число опубликованных TMA-проектов/нед — прямое доказательство, что фича востребована.

## 3. Флаги — дисциплина STUDIO_V4

Зеркалим паттерн V3 (`settings.STUDIO_V3`), но **по-фазно** — каждая фаза за своим под-флагом, по умолчанию **off** в проде до прогона. Это и есть условие «каждая фаза независимо деплоима».

| Флаг (env → settings) | Фаза | Покрывает |
|-----------------------|------|-----------|
| `STUDIO_V4_TOKEN_BILLING` | 0 | Биллинг по факту, учёт input-токенов, контекст-кэш COMMITS.md |
| `STUDIO_V4_PROVIDER_FALLBACK` | 0 | Вторичный клиент в `get_client` |
| `STUDIO_V4_AUTOFIX` | 0 | Рантайм-петля: build+guardian после фикса, dedup, лимит |
| `STUDIO_V4_STREAMING` | 1 | Стрим `file_delta` во фронт |
| `STUDIO_V4_GUARDIAN_CONTEXT` | 1 | Карта проекта в Guardian |
| `STUDIO_V4_RU_STACK` | 2 | RU-шаблоны, Selectel/Timeweb-деплой, GigaChat |
| `STUDIO_V4_TMA` | 3 | Telegram Mini App стек целиком |

**Правила дисциплины:**
- Под-флаги независимы: можно включить Ф0 в проде, оставив Ф1-3 выключенными.
- Каждый под-флаг проверяется в точке ветвления (`tasks.py`, `base.py`, агенты) — `if getattr(settings, 'STUDIO_V4_…', False)`.
- Метрики (Ф0) пишутся **всегда**, даже при выключенных флагах — чтобы измерять baseline до раскатки.
- Откат любой фазы = выключение флага без отката кода.

## Приложение C — Переменные окружения V4: что каждая делает

### Фича-флаги (0 = выключено, 1 = включено)

| Переменная | Что включает |
|---|---|
| `STUDIO_V4_TOKEN_BILLING=1` | Списание звёзд по реальному числу токенов (prompt + completion), а не по фиксированным константам. Точный биллинг — пользователь платит ровно за то, что сгенерировано. |
| `STUDIO_V4_COMMITS_CACHE=1` | В промпт coder-агента вместо полного COMMITS.md вставляются только заголовки шагов. Сокращает prompt_tokens на длинных проектах, снижает стоимость генерации. |
| `STUDIO_V4_PROVIDER_FALLBACK=1` | При ошибке 5xx или таймауте от основного провайдера (laozhang.ai) автоматически повторяет запрос на резервный URL (`LAOZHANG_API_URL_FALLBACK`). Повышает надёжность пайплайна. |
| `STUDIO_V4_AUTOFIX=1` | Ошибки из консоли iframe (JavaScript-ошибки в превью) автоматически отправляются в coder для исправления. Дубли отсекаются по SHA256-хэшу. После `STUDIO_MAX_AUTOFIX` уникальных ошибок пайплайн ставится на паузу. |
| `STUDIO_V4_STREAMING=1` | Живой стриминг кода в редактор пока coder его пишет (как в bolt.new). Пользователь видит файл появляющимся токен за токеном. После каждого шага iframe автоматически перезагружается. |
| `STUDIO_V4_GUARDIAN_CONTEXT=1` | Guardian-агент получает карту экспортов всего проекта (какие функции/классы в каких файлах). Лучше находит несуществующие импорты и битые ссылки между файлами. |
| `STUDIO_V4_RU_STACK=1` | Включает российские интеграции: Робокасса, VK ID, Яндекс.Карты, Telegram Login. Architect добавляет RU-правила в DESIGN.md (даты ДД.ММ.ГГГГ, суммы в ₽, 152-ФЗ). Появляется деплой на Timeweb и Selectel. GigaChat доступен как модель. |
| `STUDIO_V4_TMA=1` | Включает генерацию Telegram Mini App. Новый тип стека `tma` в Studio. Scaffold: Vite + React + @twa-dev/sdk. Превью работает через mock window.Telegram.WebApp. Платежи через Telegram Stars. |
| `STUDIO_MAX_AUTOFIX=3` | Максимум уникальных ошибок консоли, после которых autofix останавливается и ждёт человека. По умолчанию 3. |

## Приложение B — Что осознанно НЕ делаем

- **WebContainers / запуск Node в браузере** — чужой непреодолимый ров, сжигание ресурсов. Остаёмся на Docker-sandbox.
- **Переписывание pipeline** — Interviewer→Architect→Coder→Guardian, FILE_BLOCKS, EDIT-блоки, валидаторы, continuation, эскалация, sandbox, Gitea — работают, V4 строится поверх.
- **Нативные мобильные приложения** — TMA закрывает «мобильный» сценарий для РФ-аудитории дешевле и без сторов.
- **Автоматизация чата BotFather** — отдаём пользователю инструкцию, не имитируем человека в боте.
